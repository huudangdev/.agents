# Marcus Fleet V30.0 Release Notes

## Focus

V30.0 upgrades Marcus Fleet from a strong multi-agent execution framework into a
more governable planning and research system. The release is additive: existing
slash commands and legacy `/planning` output files remain valid.

## Major Changes

- Added Spec Kit-inspired feature lifecycle:
  - `.agents/memory/constitution.md`
  - `.agents/specs/<feature-id>/`
  - `.agents/templates/*`
  - `.agents/scripts/create_feature_spec.py`
  - `.agents/scripts/validate_specs.py`
  - `/marcus.specify`, `/marcus.clarify`, `/marcus.plan`, `/marcus.tasks`, `/marcus.verify`
- Upgraded `/planning` to V30 deep research:
  - Preserves eight legacy `/docs` outputs.
  - Adds `/docs/research/sources.jsonl`, `evidence.jsonl`, `claims.jsonl`,
    `contradictions.md`, and `research_manifest.json`.
  - Adds claim/evidence/contradiction validation through
    `.agents/scripts/validate_planning_research.py`.
- Hardened sandbox execution:
  - Removed `eval`.
  - Rejects shell metacharacter chaining.
  - Uses an explicit allow-list for bounded commands.
- Unified TrustGraph runtime configuration:
  - `.agents/trustgraph.env.example`
  - `.agents/adapters/trustgraph_config.py`
  - Shared Neo4j/Chroma settings across adapters and viewer API.
- Improved TrustGraph viewer:
  - Added `/api/health`.
  - Added live runtime status UI.
  - Removed explicit `any` typing from key graph components.
  - Replaced shell string execution in Chroma route with `execFile`.
- Updated CI template:
  - Adds spec validation.
  - Runs planning research validation when `/docs/research/` exists.
  - Calls incremental TrustGraph sync with explicit changed files.

## V30.1 Addendum: Develop Knowledge Ledger

- Upgraded `/develop` so code-phase agents must create implementation memory,
  not only source code.
- Added `/docs/development/` artifact contract:
  - `development_manifest.json`
  - `index.md`
  - `epics/*.md`
  - `modules/*.md`
  - `features/*.md`
  - `pages/*.md`
  - `tasks/*.md`
- Added templates for each development artifact bucket.
- Added `.agents/scripts/create_development_docs.py` to scaffold the code-phase
  documentation package.
- Added `.agents/scripts/validate_development_docs.py` to block incomplete
  code-phase documentation.
- Updated `.clinerules`, `USAGE_GUIDE.md`, and `/develop` so implementation
  notes must include `owner_skill`, `source_trace`, write/code scope, and
  verification evidence.

## V30.2 Addendum: Continuous Documentation Sync

- Added a PM-grade continuity gate for long POC builds where `/develop` repeats
  many code/test cycles.
- Added `/docs/development/sync/` as an append-only documentation sync ledger.
- Development scaffolding creates `sync_manifest.json`; individual sync notes are
  created only when a real code slice changes source files.
- Added `.agents/scripts/create_doc_sync_note.py` to create per-code-slice sync
  notes from explicit changed files or `git diff`.
- Added `.agents/scripts/validate_doc_sync.py` to fail when source files changed
  without a sync note and documentation review.
- Updated `/develop` with Node 6.5 Doc Sync Checkpoint.
- Updated `/quick_fix` so behavior-changing hotfixes also create sync notes and
  run doc-sync validation.
- Updated `.clinerules`, README, usage guide, CI template, and enterprise
  architecture so docs are updated by append or targeted patch, not wholesale
  replacement.

## V30.3 Addendum: Substantive Documentation Quality

- Added `.agents/DEVELOPMENT_DOCS_QUALITY_RUBRIC.md`.
- Strengthened `validate_development_docs.py` to reject template residue,
  unchecked boxes, `TBD`, `pending`, shallow notes, missing rationale, and
  missing concrete code paths.
- Strengthened `validate_doc_sync.py` to reject shallow sync notes and generic
  unfinished decisions.
- Upgraded development templates with PM notes, implementation commentary,
  change logs, rationale, evidence, risk, and quality-bar instructions.

## V30.4 Addendum: Mermaid and Global Docs Enforcement

- Development docs now require Mermaid diagrams for epic, module, feature, page,
  and task notes.
- `validate_development_docs.py` rejects development notes without a Mermaid
  code fence.
- `validate_doc_sync.py` now requires at least one global planning doc under
  `/docs` to be updated when source behavior changes.
- Sync notes must include at least one `updated because` decision for a global
  doc.
- Templates now include Mermaid sections and diagram-type guidance.

## V31.0 Addendum: Epic-First Development Ledger

- New `/develop` ledgers now default to an epic-first source-of-truth topology:
  `docs/development/E-001-description/{epic.md,features,modules,pages,tasks,sync}`.
- Canonical IDs encode hierarchy:
  - `E-001-description`
  - `F-001-001-description`
  - `M-001-001-description`
  - `P-001-001-description`
  - `T-001-001-001-description`
- `create_development_docs.py` now scaffolds V31 epic-first ledgers by default
  for new projects while preserving legacy-flat mode for existing manifests.
- `validate_development_docs.py` now validates V31 parent-child topology,
  canonical filenames, exact frontmatter IDs, `parent_epic`, unexpected root
  artifacts, legacy flat buckets inside V31 ledgers, and placeholder `*-000`
  IDs.
- `create_doc_sync_note.py` supports `--epic-id` for epic-local sync notes, and
  `validate_doc_sync.py` reads both root sync notes and `E-###-*/sync/*.md`.
- Added the `development-ledger-architect` skill to route docs creation,
  migration, validation, and archive decisions through the V31 ontology.
- Legacy flat ledgers remain supported when their manifest has not opted into
  `topology: epic_first`; migration should archive duplicates rather than
  deleting history silently.

## V31.1 Addendum: Product Governance and Docs-Before-Code

- `/develop` now requires docs to be updated or confirmed before material code
  edits. Sync notes must include `## Docs Before Code` evidence.
- Every epic/module/feature/page/task note now requires Jira-style `Story`,
  `Priority`, `Relationship Map`, and `Work Log` sections.
- Every epic now requires an `Issues` table for QA-detected or QA-reviewed
  issues with source, priority, status, owner, evidence, and resolution.
- Relationship maps must use explicit labels: `DEPENDS_ON`, `BLOCKS`,
  `ENABLES`, `IMPLEMENTS`, `USES`, `EXTENDS`, `CONFLICTS_WITH`, `SUPERSEDES`,
  `DUPLICATES`, and `RELATES_TO`.
- `validate_development_docs.py` rejects missing Story/Priority/relationship
  labels/Work Log/epic Issues. `validate_doc_sync.py` rejects missing
  docs-before-code evidence in sync notes.

## V31.2 Addendum: Brownfield Doc Reconciliation Slash Command

- Added `/doc_reconcile` for active projects whose docs are stale, flat,
  duplicated, shallow, or not aligned with code.
- Added `audit_development_docs.py` to inventory code/docs before reconciliation
  and emit `docs/development/audits/*-code-docs-audit.md`.
- Added a dedicated `issues.md` template for every epic. V31 epic-first
  validation now requires `issues.md` with `id: ISSUES-<epic-id>`.
- `/doc_reconcile` requires code-derived mapping to epics/features/modules/pages
  and tasks, labeled relationships, Jira Story/Priority, Mermaid, QA issues,
  global docs updates, and strict validation before returning.
- Governance hardening follow-up: brownfield projects with missing planning
  docs, boilerplate-only docs, absent `docs/development/`, or template-only
  ledgers now treat `/doc_reconcile` as a blocking prerequisite before
  behavior-changing `/develop`, `/quick_fix`, or implementation skill work.
- Extended hard-gate coverage:
  - `/refactor-planning` now has an explicit brownfield doc readiness gate and
    must route to `/doc_reconcile` before refactor planning when docs are not
    ready.
  - `/quick_fix` is now documented as a bypass only for doc-ready projects; it
    must fail closed into `/doc_reconcile` on unreconciled brownfield work.
  - `/bootstrap` guidance now includes `/doc_reconcile` as the correct next
    route for existing brownfield projects whose docs package is not ready.
  - GitLab CI template now fails not only when `docs/development/` is missing
    on source-changing diffs, but also when the required planning package files
    are missing.

## Compatibility

- `/planning` still writes:
  - `/docs/prd.md`
  - `/docs/tasks.md`
  - `/docs/knowledge.md`
  - `/docs/decisions.md`
  - `/docs/memory.md`
  - `/docs/planning/flows.md`
  - `/docs/planning/screens.md`
  - `/docs/planning/diagrams.md`
- `/design` and `/develop` continue to use approved `/docs` artifacts.
- `/quick_fix` remains the correct route for small localized fixes only when
  the target project is already doc-ready under the brownfield governance gate.
- TrustGraph adapters still degrade gracefully when Neo4j/Chroma are offline.
