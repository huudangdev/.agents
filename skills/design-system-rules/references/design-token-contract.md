# Design System Contract

Use this skill when token discipline or reusable UI structure matters.

## Required Inputs

- Current design tokens and primitives
- Active feature docs and component context
- The reuse or consistency problem being solved

## Decision Rules

- Prefer existing tokens and primitives over new abstraction layers.
- Avoid magic numbers and untracked styling drift.
- Keep accessibility and readability mandatory.

## Output Contract

- State the token or component rule.
- Identify where it applies and why.
- Describe how the rule will be verified in implementation.
