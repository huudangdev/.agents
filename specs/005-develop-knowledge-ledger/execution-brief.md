## 1. Operator Intent Snapshot

## 1. Purpose

Ensure `/develop` preserves implementation knowledge at the same granularity as
the code being produced. When agents enter code phase, they must create separate
documentation for epics, modules, tasks, features, and pages so future agents can
recover ownership, reasoning, verification, and handoff context without
re-running planning.


## 2. Required Behavior

## 3. Functional Requirements

- `FR-001`: `/develop` MUST require a `/docs/development/` knowledge ledger for
  material implementation work.
- `FR-002`: The ledger MUST include `development_manifest.json`, `index.md`,
  and bucket directories for `epics`, `modules`, `features`, `pages`, and
  `tasks`.
- `FR-003`: The system MUST provide templates for each bucket.
- `FR-004`: The system MUST provide a scaffold script that creates the ledger
  without replacing existing files by default.
- `FR-005`: The system MUST provide a validator that checks frontmatter,
  owner skill, source trace, verification, and code/write scope.
- `FR-006`: CI documentation guidance MUST validate development docs when
  `/docs/development/` exists.


## 5. Acceptance Criteria

- `AC-001`: Given a developer invokes `/develop`, when behavior is materially
  implemented, then `/docs/development/` is initialized or updated before final
  completion.
- `AC-002`: Given `/docs/development/` exists, when
  `validate_development_docs.py --strict-counts` runs, then missing bucket docs
  or required metadata fail validation.
- `AC-003`: Given backend-only work has no page changes, when the manifest sets
  page `minimum_count` to `0`, then validation can pass while `index.md`
  documents why page notes are absent.
- `AC-004`: Given existing planning docs exist, when the development ledger is
  created, then the approved planning package is not renamed or collapsed.
