# Feature Specification: Develop Knowledge Ledger

> Feature ID: `005-develop-knowledge-ledger`
> Created: `2026-04-21`
> Status: Draft
> Source Prompt: Upgrade /develop so code-phase agents create separate documentation files for epics, modules, tasks, features, and pages during implementation.

## 1. Purpose

Ensure `/develop` preserves implementation knowledge at the same granularity as
the code being produced. When agents enter code phase, they must create separate
documentation for epics, modules, tasks, features, and pages so future agents can
recover ownership, reasoning, verification, and handoff context without
re-running planning.

## 2. User Stories

- [x] As an operator, I need `/develop` to create code-phase docs per epic,
      module, feature, page, and task so implementation knowledge does not
      disappear into source diffs only.
- [x] As a future agent, I need each implementation note to name owner skill,
      source trace, code scope, and verification so I can continue work safely.
- [x] As a reviewer, I need a validator to detect missing or shallow
      development docs before code-phase completion is reported.

## 3. Functional Requirements

- `FR-001`: `/develop` MUST require a `/docs/development/` knowledge ledger for
  material implementation work.
- `FR-002`: The ledger MUST include `development_manifest.json`, `index.md`,
  and bucket directories for `epics`, `modules`, `features`, `pages`, and
  `tasks`.
- `FR-003`: The system MUST provide templates for each bucket.
- `FR-004`: The system MUST provide a scaffold script that creates the ledger
  without replacing existing files by default.
- `FR-005`: The system MUST provide a validator that checks frontmatter,
  owner skill, source trace, verification, and code/write scope.
- `FR-006`: CI documentation guidance MUST validate development docs when
  `/docs/development/` exists.

## 4. Non-Functional Requirements

- `NFR-001`: Maintainability: docs must be bucketed and predictable so agents can
  locate implementation memory cheaply.
- `NFR-002`: Compatibility: legacy `/planning`, `/design`, and `/develop`
  handoffs must remain intact.
- `NFR-003`: Safety: scaffold must not overwrite existing development docs
  unless explicitly forced.
- `NFR-004`: Auditability: every artifact must include source trace and
  verification evidence.

## 5. Acceptance Criteria

- `AC-001`: Given a developer invokes `/develop`, when behavior is materially
  implemented, then `/docs/development/` is initialized or updated before final
  completion.
- `AC-002`: Given `/docs/development/` exists, when
  `validate_development_docs.py --strict-counts` runs, then missing bucket docs
  or required metadata fail validation.
- `AC-003`: Given backend-only work has no page changes, when the manifest sets
  page `minimum_count` to `0`, then validation can pass while `index.md`
  documents why page notes are absent.
- `AC-004`: Given existing planning docs exist, when the development ledger is
  created, then the approved planning package is not renamed or collapsed.

## 6. Clarifications

- No unresolved clarifications. User explicitly requested separate code-phase
  documentation for epics, modules, tasks, features, and pages.

## 7. Constraints

- Constitution articles that apply: Context Before Construction, Evidence Before
  Completion, Human-Visible Handoff.
- Existing files or modules in scope: `/develop`, `.clinerules`, README, usage
  guide, release notes, templates, scripts, CI template.
- Files or modules out of scope: application source code generated by future
  product projects.
- Compatibility requirements: additive only; do not replace legacy `/docs`
  planning outputs.

## 8. Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Agents skip docs and only write code | Future context loss | Validator and workflow completion gate |
| Docs become generic boilerplate | Low-quality handoff | Required source trace, owner skill, code scope, verification sections |
| Backend-only tasks fail because pages are absent | False validation failure | Manifest supports `pages.minimum_count = 0` with index explanation |
